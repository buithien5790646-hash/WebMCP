## 1. 项目概览 (WebMCP)

**目标**: 通过 MCP 协议，将基于 Web 的 AI (ChatGPT/DeepSeek) 连接到本地 VS Code 环境，赋予其文件读写、代码执行和 Git 管理能力。

**核心架构**: `Browser AI` <-> `Browser Extension (Bridge)` <-> `VS Code Extension (Gateway)` <-> `Local Tools`

---

## 2. 工程架构 (Engineering)

### 🏗️ Monorepo 结构 (pnpm workspace)
项目采用 Monorepo 模式管理，根目录包含 `pnpm-workspace.yaml`。

- **根目录**: 统一依赖管理，共享配置。
- **`mcp-core/`**: (Refactored) 核心业务逻辑，完全解耦的 MCP Gateway 实现。从 VS Code 插件中剥离，现位于根目录。
- **`mcp-gateway-desktop/`**: (New) 桌面端主程序 (Electron + React)。
- **`shared/`**: 存放两端共用的类型定义 (`types.ts`) 和协议常量，确保通信一致性。
- **`mcp-gateway-vscode/`**: VS Code 插件项目 (Legacy Entry, Maintenance Mode)。
- **`mcp-bridge-browser/`**: 浏览器插件项目 (Vite + CRXJS + TS)。

## 3. 组件详解

### 🔌 A. VS Code 插件 (`mcp-gateway-vscode`)
**角色**: MCP Server / Host (聚合器 & 网关)

- **核心功能**:
  - **本地服务器**: 启动 Express Server (默认端口 `34567`)。
    - **端口自增**: 若端口被占用，自动 +1 重试（最大 20 次），支持多实例并存。
  - **工具聚合**: 管理并启动子 MCP 服务（如 `@modelcontextprotocol/server-filesystem`, `git`, `fetch`），将它们的能力合并暴露。
  - **安全握手**: 
    - 每次启动生成随机 `UUID Token`。
    - 提供 `/bridge` 中转页，自动携带 Token 和端口信息唤起浏览器，实现零配置连接。
  - **配置管理**: 通过 `settings.json` 配置子服务参数（如 GitHub Token、Docker 容器参数）。

### 🧩 B. 浏览器插件 (`mcp-bridge-browser`)
**角色**: MCP Client / Bridge (注入器 & 客户端)

#### 1. 核心逻辑 (Content Scripts)
- **模块化架构 (src/)**:
  - `content/main.ts`: **大脑**。负责轮询 DOM 消息、解析 AI 响应、调度任务队列。
  - `modules/utils.ts`: **基建**。提供悬浮日志 (Logger)、多语言 (i18n) 字典支持。
  - `modules/ui.ts`: **交互**。实现 HITL 审批弹窗 (Shadow DOM)、自动发送、输入框回填 (保持原生 DOM 操作以减少体积)。
  - `modules/config.ts`: **配置**。集中管理各 AI 平台的 DOM 选择器。
- **持久化**: `background/index.ts` 负责在插件更新时保护用户配置不被覆盖，并处理安全熔断。

### 🖥️ C. Desktop App (`mcp-gateway-desktop`)
**角色**: 新一代图形化管理端 (The Future Host)
- **Tech Stack**: Electron, Vite, React, TypeScript.
- **核心功能**:
  - **Dashboard**: 可视化管理多个 Gateway 实例，查看实时日志。
  - **Library**: 管理 MCP Server 定义（支持添加/移除 stdio/sse 服务）。
  - **Auth**: 自动生成 Bridge URL 和 Token。
  - **Persistence**: 使用 `electron-store` 本地持久化配置。

## 3. 关键特性机制 (Shared)

### 🛡️ Human-in-the-Loop (HITL) 人工审批 (v2.1 Secure & Localized)
- **机制**: 
  - **拦截**: `content.js` 在执行工具前检查 `protected_tools` 名单。
  - **自动防御 (Auto-Protect)**: 首次发现新工具时，默认自动加入保护名单。
- **安全与交互 (Security & UX)**:
  - **防注入 (Sanitization)**: 对工具参数进行 HTML 转义，防止 XSS 攻击和布局崩坏。
  - **视图切换 (Inline View)**: 永久授权采用卡片内平滑切换，替代原生 Alert，体验统一。
  - **国际化 (I18n)**: 弹窗 UI 完全支持中/英双语自适应。
  - **Shadow DOM**: 样式隔离，互不干扰。

### ⚡ 并发批处理队列 (Concurrency Queue)
- **痛点**: 解决 AI 一次性调用多个工具导致的回填乱序或漏发问题。
- **逻辑**: `resultBuffer` 暂存所有结果 -> 等待当前轮次 **所有** 工具完成 -> 按 DOM 顺序拼接 -> 一次性回填。

### 📉 工具分层发现 (Tiered Tool Discovery)
- **机制**: 网关将工具分为 **Hot (基础)** 和 **Cold (扩展)** 两类。
- **流程**: 
  1. `list_tools`: Hot 工具返回全量 Schema；Cold 工具返回 `[Schema Hidden]` 描述。
  2. **Query**: AI 识别到隐藏标记，调用 `get_tool_definitions(names)`。
  3. **Execute**: 获得完整定义后，AI 发起真正的工具调用。
